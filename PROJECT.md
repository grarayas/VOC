# VOC Platform вЂ” Project Documentation

> **Context**: Final Year Project (PFE) вЂ” Refactoring of a Vulnerability Operation Center platform built inside a banking enterprise IT environment.

---

## 1. Project Overview

The VOC Platform is a centralized web application for managing IT vulnerabilities across a banking organization. It replaces a legacy system built with scattered Python scripts, PowerShell automation, and JSON flat files вЂ” migrating everything into a structured, scalable Flask + SQL Server stack.

### Core Problem Solved

The old system had no central database, no access control by team, no proper lifecycle tracking, and no scalable reporting. Vulnerability data lived in `tenable_cmdb_match.json` files generated by ad-hoc scripts. The refactoring centralizes all of this into a proper relational database with a web interface, role-based access, and structured ETL pipelines.

## 4. Database

**SQL Server database name**: `VOC_DataBase`

**Connection**: Windows Authentication (Trusted Connection) in development. The connection string is built in `config.py`:

```python
SQLALCHEMY_DATABASE_URI = (
    "mssql+pyodbc://@localhost/VOC_DataBase"
    "?driver=ODBC+Driver+17+for+SQL+Server"
    "&Trusted_Connection=yes"
)
```

**8 tables** вЂ” see `MODELS.md` for full detail.

### Key Design Decisions

- `asset_vulnerabilities` is the **central table** вЂ” it replaces the old `tenable_cmdb_match.json`. Every vulnerability occurrence (one vuln on one asset) is a row here.
- `vulnerabilities` holds **pure Tenable data** (CVE, CVSS, plugin info) вЂ” independent of the asset.
- `vuln_history` is a **unified activity log** вЂ” both status transitions and comments are stored here, differentiated by `entry_type` (`status_change` | `comment`).
- `sla_tracking` is **1:1 with asset_vulnerabilities** вЂ” every occurrence has exactly one SLA record.
- `scopes` has a `category` field: `applicatif` | `systeme` | `systeme_app`.

## 5. Authentication & Access Control

### Active Directory

Authentication uses LDAP3 to bind against an Active Directory server. In development mode (`FLASK_DEBUG=1`), if AD is unreachable, a local fallback accepts any username that exists in the `users` table regardless of password.

```python
# utils/auth.py
def authenticate_ad(username, password) -> bool
def authenticate_local(username, password) -> bool   # dev fallback only
def get_user_from_db(username) -> User | None
```

### Role-Based Access Control (RBAC)

Two roles: `admin` and `user`.

- `admin` вЂ” sees all scopes, all data, all admin functions
- `user` вЂ” sees only data linked to their `scope_id`

### Scope Filtering

Every data query is filtered through `scope_filter()`:

```python
# utils/auth.py
def scope_filter(query, model_scope_field):
    if current_user.is_admin:
        return query
    return query.filter(model_scope_field == current_user.scope_id)
```

Usage in routes:
```python
q = AssetVulnerability.query.join(Asset)
q = scope_filter(q, Asset.scope_id)
```

### Decorators

```python
@admin_required       # 403 if not admin
@login_required       # redirect to login if not authenticated (Flask-Login)
```

## 7. Vulnerability Lifecycle

Each vulnerability occurrence (`asset_vulnerabilities` row) has a `vuln_status` field and a `branch` field.

### Two workflow branches

**Standard branch** (`branch = 'standard'`):
```
Open в†’ In Progress в†’ Fixed в†’ Closed
```

**Risk Assessment branch** (`branch = 'risk_assessment'`):
```
Open в†’ In Progress в†’ Risk Assessment в†’ Accepted Risk
                                            в†“
                                    (stays active, can reopen)
                                            в†“
                                        In Progress
```

### Status rules

- `Closed` is the **only terminal state** вЂ” the vuln disappears from V-Tracker
- `Accepted Risk` means the vuln cannot be fixed now (budget, vendor dependency) вЂ” it stays **visible and active**
- `Accepted Risk` can only be set by an **admin**
- `branch` is `NULL` when status is `Open` (path not yet decided)

### Allowed transitions (model property)

```python
AssetVulnerability.next_statuses
# Returns list of allowed next statuses from current state
```

## 8. Modules

### V-Tracker (`/vtracker`)

Core module. Routes:

| Route | Method | Description |
|---|---|---|
| `/vtracker/` | GET | Dashboard with KPIs (total, critical, open, breached SLA) |
| `/vtracker/list` | GET | Paginated list with filters (severity, status, source, search) |
| `/vtracker/detail/<av_id>` | GET | Full vulnerability detail + activity history |
| `/vtracker/status/<av_id>` | POST | Change status + optional comment |
| `/vtracker/comment/<av_id>` | POST | Add comment to history |

Filters on list view: `severity`, `status`, `source`, `search` (searches plugin_name, CVE, asset name, IP).

### V-Hub (`/vhub`)

Manual vulnerability input for 3 sources:

| Route | Source | Description |
|---|---|---|
| `/vhub/cti` | `cti` | Cyber Threat Intelligence input |
| `/vhub/bitsight` | `bitsight` | Third-party risk score input |
| `/vhub/redteam` | `redteam` | Penetration test finding input |

All three share the same `_create_vuln_entry()` logic:
1. Build `vuln_id` from `IP + "PL" + plugin_id + "PO" + port`
2. Create or retrieve `Vulnerability` record
3. Match asset by IP in `assets` table
4. Check for duplicate (same vuln_id + asset_id)
5. Create `AssetVulnerability` + `SlaTracking` records

SLA days by criticality: `Critical=15`, `High=30`, `Medium=90`, `Low=90`

### Reporting (`/reporting`)

Aggregation dashboard вЂ” data grouped by severity, status, source. Power BI connects via DirectQuery on SQL Server.

### Admin (`/admin`)

Admin-only. Manages users (activate/deactivate, change scope) and shows ETL execution logs.

## 9. ETL Pipeline

Three standalone Python scripts in `/etl/`:

| Script | Input | Output |
|---|---|---|
| `etl_cmdb.py` | `Server_Usage_List.csv` + `All_Active_CIs.csv` | `assets` table |
| `etl_tenable.py` | Tenable export CSV/XML | `vulnerabilities` + `asset_vulnerabilities` |
| `etl_scoring.py` | `asset_vulnerabilities` data | Updates `risk_score`, `likelihood`, `impact`, `asset_exposure`, `sla_tracking` |

### Correlation logic (from old platform, preserved)

Tenable data is matched to CMDB assets using: `ip_address` в†’ `netbios_name` в†’ `dns_name` (in priority order, whichever is available).

### vuln_id format

```
{ip_address}PL{plugin_id}PO{port}
# Example: 10.30.3.10PL96982PO445
```

### Risk score formula (3 dimensions)

```
risk_score = f(cvss_v3_score, asset_criticality, asset_exposure)
```

- `cvss_v3_score` вЂ” from Tenable (0вЂ“10)
- `asset_criticality` вЂ” from CMDB (`Critical=1.0`, `Important=0.7`, `Standard=0.4`)
- `asset_exposure` вЂ” calculated (0вЂ“100, based on network zone, environment)

## 12. Important Conventions

- All routes are protected by `@login_required`
- All data queries go through `scope_filter()` вЂ” never bypass it
- Status transitions are validated through `AssetVulnerability.next_statuses` вЂ” never set `vuln_status` directly without this check
- Every status change must create a `VulnHistory` row with `entry_type='status_change'`
- Every comment must create a `VulnHistory` row with `entry_type='comment'`
- `Accepted Risk` transitions require admin validation вЂ” enforce via role check in route
- `vuln_status != 'Closed'` filter must be applied on all V-Tracker queries (closed vulns are hidden)
