## 1. Module Auth

Le module Auth gère l'ensemble du cycle d'authentification : connexion Active Directory, génération de tokens JWT, protection des routes et contrôle d'accès basé sur les rôles et les scopes.

### 1.1 Structure des fichiers

| Fichier | Rôle |
|---|---|
| `blueprints/auth/__init__.py` | Déclaration du Blueprint Flask `auth` |
| `blueprints/auth/routes.py` | 4 endpoints HTTP (login, logout, me, page) |
| `blueprints/auth/services.py` | Logique métier : AD, scope, user, token |
| `models/user.py` | Modèle SQLAlchemy User |
| `utils/auth.py` | Fonction `scope_filter()` pour les requêtes DB |
| `utils/decorators.py` | Décorateurs `@admin_required` et `@scope_required` |
| `templates/auth/login.html` | Page de connexion (SPA-like avec fetch API) |

---

### 1.2 Endpoints HTTP

| Méthode | Route | Description |
|---|---|---|
| `GET` | `/auth/login` | Affiche la page HTML de connexion |
| `POST` | `/api/auth/login` | Authentifie l'utilisateur (AD + JWT) |
| `GET` | `/api/auth/me` | Retourne l'identité du token courant |
| `POST` | `/api/auth/logout` | Invalide le cookie JWT côté serveur |

---

### 1.3 Flux d'authentification — `POST /api/auth/login`

Le flux suit **7 étapes séquentielles** :

```
1. Validation des champs         → username et password non vides
2. authenticate_ad()             → double bind LDAP pour vérifier les credentials
3. map_ad_groups_to_scope()      → résolution du scope depuis les groupes AD
4. get_or_create_user()          → provisioning automatique en base
5. Vérification is_active        → compte désactivé = 403
6. generate_token()              → création du JWT signé avec claims custom
7. Réponse                       → token en JSON + cookie HttpOnly (set_access_cookies)
```

---

### 1.4 Service : `authenticate_ad()`

Authentification en **deux binds LDAP successifs** :

```
Bind 1 — Service account (lecture seule)
  Connexion avec AD_BIND_DN / AD_BIND_PWD
  Recherche utilisateur par sAMAccountName
  Attributs récupérés : mail, givenName, sn, memberOf, distinguishedName

Bind 2 — Vérification mot de passe
  Re-connexion avec le DN de l'utilisateur et son mot de passe
  Si bind échoue → retourne None (credentials invalides)

Protection : escape_filter_chars() sur le username (anti-injection LDAP)
Gestion erreurs : try/except global → None sans exposer les détails
```

---

### 1.5 Service : `map_ad_groups_to_scope()`

Mappe les groupes AD de l'utilisateur vers un scope en base. **Le premier groupe matché gagne.**

| Groupe AD | Scope résolu |
|---|---|
| `VOC_ADMIN` | Cyber Security |
| `VOC_WINDOWS` | Windows Team |
| `VOC_LINUX` | Linux Team |
| `VOC_NETWORK` | Network Team |
| `VOC_DATACENTER` | Datacenter Team |
| `VOC_MIDDLEWARE` | Middleware & Tools |
| `Admins du domaine` | Cyber Security *(lab demo)* |

---

### 1.6 Service : `get_or_create_user()`

Auto-provisioning : les utilisateurs ne sont **pas créés manuellement**.

```
Si user existe en DB :
  → Synchroniser email si changé dans l'AD
  → Retourner l'user existant

Si user n'existe pas :
  → role = "admin" si groupe VOC_ADMIN présent, sinon "user"
  → Créer User(username, email, role, scope_id, is_active=True)
  → db.session.commit()
```

---

### 1.7 Service : `generate_token()`

Crée un JWT signé via Flask-JWT-Extended. Claims inclus dans le token :

| Claim | Valeur |
|---|---|
| `identity` | `user_id` (string) — identifiant principal |
| `username` | sAMAccountName de l'AD |
| `role` | `"admin"` ou `"user"` |
| `scope_id` | FK vers la table scopes |
| `scope` | Nom du scope (ex: `"Windows Team"`) |
| `is_admin` | Boolean précalculé depuis `role == "admin"` |

> Les claims sont lus à chaque requête via `get_jwt()` **sans aller en base**.

---

### 1.8 Décorateurs de protection — `utils/decorators.py`

#### `@admin_required`
Bloque l'accès avec **403** si `is_admin = False` dans les claims JWT.

#### `@scope_required(*allowed_scopes)`
Vérifie que le scope de l'utilisateur est dans la liste des scopes autorisés.  
**Les admins bypassent toujours cette vérification.**

```python
# Exemple d'utilisation
@scope_required("Windows Team", "Linux Team")
def ma_route():
    ...
```

---

### 1.9 Utilitaire : `scope_filter()` — `utils/auth.py`

Filtre de visibilité appliqué aux requêtes SQLAlchemy. Garantit l'isolation des données par scope.

```python
def scope_filter(query, model_scope_field):
    claims = get_jwt()
    if claims.get('is_admin'):
        return query                                         # admin voit tout
    return query.filter(
        model_scope_field == claims.get('scope_id')         # user → son scope uniquement
    )
```

---

### 1.10 Frontend : `login.html`

Page SPA-like utilisant `fetch()` pour appeler l'API sans rechargement.

| Étape | Action |
|---|---|
| Submit formulaire | `fetch POST /api/auth/login` avec JSON `{username, password}` |
| Réponse 200 | Stocker token + user dans localStorage, rediriger |
| Réponse 4xx | Afficher le message d'erreur dans l'alerte Bootstrap |
| Token expiré (reload) | Appel `/api/auth/me` → si 401, purger localStorage |

**Double transport du token :** cookie HttpOnly (posé par le serveur) + localStorage (accessible au JS).

## 2. Module V-Tracker

Le V-Tracker est le module central de visualisation et de suivi des vulnérabilités. Il gère l'affichage filtré par scope, le cycle de vie, les commentaires et la mise à jour des SLA.

### 2.1 Structure des fichiers

| Fichier | Rôle |
|---|---|
| `blueprints/vtracker/__init__.py` | Déclaration du Blueprint `vtracker` |
| `blueprints/vtracker/routes.py` | 5 routes : dashboard, list, detail, status, comment |
| `templates/vtracker/dashboard.html` | Vue KPI (total, open, critical, SLA breached) |
| `templates/vtracker/list.html` | Liste paginée avec filtres multi-critères |
| `templates/vtracker/detail.html` | Fiche complète d'une vulnérabilité |

---

### 2.2 Routes et endpoints

| Route | Méthode | Description |
|---|---|---|
| `GET  /vtracker/` | GET | Dashboard KPI filtré par scope |
| `GET  /vtracker/list` | GET | Liste paginée avec filtres |
| `GET  /vtracker/detail/<av_id>` | GET | Fiche détail d'une vulnérabilité |
| `POST /vtracker/status/<av_id>` | POST | Changer le statut (cycle de vie) |
| `POST /vtracker/comment/<av_id>` | POST | Ajouter un commentaire |

---

### 2.3 Fonction `_base_query()`

Fonction interne qui construit la requête SQLAlchemy de base avec isolation automatique par scope :

```
AssetVulnerability
  JOIN Asset          (pour scope_id + nom)
  JOIN Vulnerability  (pour severity, cve, plugin)
  FILTER vuln_status != 'Closed'

Si is_admin = False :
  FILTER Asset.scope_id == claims['scope_id']

Si is_admin = True :
  Aucun filtre → admin voit toutes les vulnérabilités
```

---

### 2.4 Filtres de la liste — `GET /vtracker/list`

| Paramètre URL | Effet |
|---|---|
| `severity` | Filtre par sévérité (Critical / High / Medium / Low / Info) |
| `status` | Filtre par statut du cycle de vie |
| `source` | Filtre par source (tenable / cti / bitsight / redteam) |
| `search` | Recherche ILIKE sur plugin_name, cve, asset.name, ip_address |
| `page` | Numéro de page (pagination 25 éléments par page) |

---

### 2.5 Cycle de vie des vulnérabilités

Deux branches coexistent selon le chemin de traitement :

**Branche Standard (remédiation) :**
```
Open  →  In Progress  →  Fixed  →  Closed
```

**Branche Risk Assessment (risque accepté) :**
```
Open  →  In Progress  →  Risk Assessment  →  Accepted Risk  →  Closed
```

Transitions autorisées (définies dans `AssetVulnerability._TRANSITIONS`) :

| Statut actuel | Transitions autorisées |
|---|---|
| `Open` | In Progress |
| `In Progress` | Fixed, Risk Assessment |
| `Risk Assessment` | Accepted Risk, In Progress |
| `Accepted Risk` | In Progress |
| `Fixed` | Closed |
| `Closed` | Aucune *(terminal)* |

> **Restriction :** seul un admin peut valider le statut `Accepted Risk`.

---

### 2.6 Route `change_status()`

Lors de chaque changement de statut, le système effectue **3 actions** :

1. Valider la transition via `av.next_statuses`
2. Mettre à jour `av.vuln_status` et `av.updated_at`
3. Enregistrer un `VulnHistory` de type `"status_change"` (audit trail)

**Actions complémentaires :**
- Premier passage hors de `Open` → définit `av.branch` (`standard` ou `risk_assessment`)
- Statut `Fixed` ou `Closed` → `sla.sla_status = "Completed"`
- Statut `Accepted Risk` → `sla.risk_acceptance = True`

---

### 2.7 Gestion des commentaires

La route `POST /vtracker/comment/<av_id>` ajoute un `VulnHistory` de type `"comment"`.  
Le commentaire est lié à l'utilisateur connecté (`user_id` depuis `claims["sub"]`) et horodaté en UTC.

## 3. Modèles SQLAlchemy

L'application utilise **7 modèles** SQLAlchemy mappant les tables SQL Server. Tous héritent de `db.Model` via Flask-SQLAlchemy.

### 3.1 `User`

| Colonne | Type / Contrainte |
|---|---|
| `user_id` | Integer PK autoincrement |
| `username` | String(100) UNIQUE NOT NULL — sAMAccountName AD |
| `email` | String(150) UNIQUE NOT NULL — synchronisé depuis l'AD |
| `role` | String(50) NOT NULL — `"admin"` \| `"user"` |
| `is_active` | Boolean NOT NULL default=True |
| `scope_id` | Integer FK → `scopes.scope_id` NOT NULL |

- **Propriété calculée :** `is_admin` → `role == "admin"`
- **Relations :** `sla_assignments` → `SlaTracking` | `history_entries` → `VulnHistory`

---

### 3.2 `Scope`

| Colonne | Type / Contrainte |
|---|---|
| `scope_id` | Integer PK autoincrement |
| `scope_name` | String(150) NOT NULL — nom de l'équipe |
| `category` | String(50) NOT NULL — `"applicatif"` \| `"systeme"` \| `"systeme_app"` |
| `bv_name` | String(150) nullable — nom BV pour matching CMDB |
| `description` | String(255) nullable |

> 20 scopes en base : **14 applicatifs**, **5 système**, **1 systeme_app**.

---

### 3.3 `Asset`

| Colonne | Type / Rôle |
|---|---|
| `asset_id` | Integer PK |
| `name` | String(150) NOT NULL — hostname principal |
| `ip_address` | String(50) — clé de corrélation CMDB/Tenable |
| `operating_system` | String(150) — OS de l'asset |
| `criticality` | String(50) — `Critical` / `Important` / `Standard` |
| `environment` | String(100) — `Production` / `Staging` / `Development` |
| `scope_id` | FK → scopes — équipe propriétaire |
| `last_synced_at` | DateTime — dernière synchronisation ETL CMDB |

- **Relation :** `occurrences` → `AssetVulnerability` (toutes les vulns liées)

---

### 3.4 `Vulnerability`

| Colonne | Type / Rôle |
|---|---|
| `vuln_id` | String(200) PK — format: `{ip}PL{plugin_id}PO{port}` |
| `plugin_id` | String(50) — identifiant Tenable |
| `plugin_name` | String(255) — nom du plugin |
| `cve` | String(100) — identifiant CVE |
| `severity` | String(50) — Critical/High/Medium/Low/Info |
| `cvss_v2_score` | Numeric(4,1) — score CVSS v2 |
| `cvss_v3_score` | Numeric(4,1) — score CVSS v3 |
| `exploitable` | Boolean — vulnérabilité exploitable ? |
| `exploit_ease` | String(100) — facilité d'exploitation |
| `steps_to_remediate` | Text — procédure de remédiation |

- **Propriété calculée :** `severity_color` → couleur Bootstrap selon sévérité

---

### 3.5 `AssetVulnerability`

Table de jointure **enrichie** entre `Asset` et `Vulnerability`. Représente une occurrence de vulnérabilité sur un asset spécifique.

| Colonne | Type / Rôle |
|---|---|
| `av_id` | Integer PK |
| `vuln_id` | String FK → `vulnerabilities` |
| `asset_id` | Integer FK → `assets` |
| `source` | String(50) — `tenable` / `cti` / `bitsight` / `redteam` |
| `vuln_status` | String(50) — statut dans le cycle de vie |
| `branch` | String(30) — `"standard"` \| `"risk_assessment"` |
| `risk_score` | Numeric(5,2) — score de risque calculé par ETL |
| `likelihood` | Numeric(5,2) — probabilité d'exploitation |
| `impact` | Numeric(5,2) — impact métier |
| `asset_exposure` | Numeric(5,2) — exposition réseau de l'asset |

- **Propriétés calculées :** `next_statuses` (transitions valides) | `status_color` (couleur Bootstrap)
- **Relations :** `sla` → `SlaTracking` (1:1) | `history` → `VulnHistory` (1:N)

---

### 3.6 `SlaTracking`

Un enregistrement SLA par `AssetVulnerability` (relation **1:1**).

| Colonne | Type / Rôle |
|---|---|
| `sla_id` | Integer PK |
| `av_id` | Integer FK → `asset_vulnerabilities` UNIQUE |
| `sla_days_target` | Integer — objectif SLA en jours |
| `sla_due_date` | Date — date limite de remédiation |
| `sla_status` | String(50) — `On Track` / `At Risk` / `Breached` / `Completed` |
| `risk_acceptance` | Boolean — risque officiellement accepté |
| `remed_responsible` | Integer FK → `users.user_id` — responsable |

> **Délais SLA par sévérité :** Critical=15j | High=30j | Medium=90j | Low=90j | Info=180j

---

### 3.7 `VulnHistory`

Journal d'audit de toutes les actions sur une `AssetVulnerability`.

| Colonne | Type / Rôle |
|---|---|
| `history_id` | Integer PK |
| `av_id` | Integer FK → `asset_vulnerabilities` |

| `entry_type` | String(30) — `"status_change"` \| `"comment"` |
| `old_status` | String(50) nullable — statut avant changement |
| `new_status` | String(50) nullable — statut après changement |
| `content` | Text nullable — texte du commentaire |
| `created_at` | DateTime UTC — horodatage |

## 4. Scripts ETL

Trois scripts Python autonomes gèrent l'alimentation de la base de données. Chaque script journalise son exécution dans `etl_logs` et utilise `pyodbc` pour communiquer directement avec SQL Server.

### 4.1 ETL CMDB — `etl_cmdb.py`

| | |
|---|---|
| **Source** | Deux fichiers CSV : `Server_Usage_List.csv` + `All_Active_CIs.csv` |
| **Cible** | Table `assets` |
| **Commande** | `python -m etl.etl_cmdb --server_usage <path> --active_cis <path>` |

#### Logique de traitement

1. Lecture et normalisation des colonnes CSV (strip, lowercase, replace espaces)
2. Fusion des deux fichiers sur `ip_address` (merge pandas, jointure LEFT)
3. Construction du cache scope : `{bv_name → scope_id}` depuis la table `scopes`
4. Pour chaque ligne : résolution du scope via `bv_name` → si absent, skip
5. Vérification d'existence par `ip_address` OU `name` dans `assets`
6. Si existant : `UPDATE` des champs ip, OS, status, environment, criticality, last_synced_at, scope_id
7. Si nouveau : `INSERT` complet de tous les champs de l'asset
8. Commit transactionnel final, rollback en cas d'exception

#### Journalisation ETL

Un enregistrement `etl_logs` est créé au début (`status="running"`) puis mis à jour à la fin avec :
`status`, `records_extracted`, `records_matched`, `records_inserted`, `records_skipped`, `error_msg`, `duration_sec`

---

### 4.2 ETL Tenable — `etl_tenable.py`

| | |
|---|---|
| **Source** | Export CSV Tenable.io |
| **Cible** | Tables `vulnerabilities` + `asset_vulnerabilities` + `sla_tracking` |
| **Commande** | `python -m etl.etl_tenable --input <tenable_export.csv>` |

#### Format de la clé primaire `vuln_id`

```
{ip_address}PL{plugin_id}PO{port}

Exemple : 10.30.3.10PL21745PO443
```

#### Logique de traitement

1. Renommage des colonnes Tenable vers le modèle interne (`TENABLE_COL_MAP`)
2. Construction du cache asset : `{ip/netbios/dns → asset_id}`
3. **Priorité de matching :** `ip_address` → `netbios_name` → `dns_name`
4. Si aucun asset correspondant : ligne ignorée (skipped)
5. Upsert `vulnerabilities` : INSERT si `vuln_id` inconnu, skip si existant
6. Upsert `asset_vulnerabilities` : si existant → UPDATE `last_observed` uniquement
7. Si nouveau AV : INSERT avec `status="Open"` + création entrée SLA automatique

#### Délais SLA automatiques à l'insertion

| Sévérité | Délai SLA |
|---|---|
| Critical | 15 jours |
| High | 30 jours |
| Medium | 90 jours |
| Low | 90 jours |
| Info | 180 jours |

---

### 4.3 ETL Scoring — `etl_scoring.py`

| | |
|---|---|
| **Source** | Données internes (`vulnerabilities` + `assets` + `sla_tracking`) |
| **Cible** | `asset_vulnerabilities` (risk_score, likelihood, impact, asset_exposure) + `sla_tracking` (sla_status) |
| **Commande** | `python -m etl.etl_scoring` |

#### Formule de calcul du Risk Score

```
crit_weight    = Critical:1.0  /  Important:0.7  /  Standard:0.4
exposure       = Production:80  /  Staging:50  /  Development:20
exploitability = 1.3 si exploitable, sinon 1.0

likelihood   = min((cvss_v3 / 10) * exploitability, 1.0)
impact       = min(crit_weight * (exposure / 100), 1.0)
risk_score   = round((likelihood*0.5 + impact*0.5) * cvss_v3, 2)
```

#### Mise à jour du statut SLA

| Condition | Nouveau statut SLA |
|---|---|
| `sla_status == "Completed"` | Inchangé (ne pas toucher) |
| `today > sla_due_date` | **Breached** |
| `(sla_due_date - today).days <= 7` | **At Risk** |
| Sinon | **On Track** |

> **Périmètre :** toutes les `AssetVulnerability` dont le statut n'est ni `"Closed"` ni `"Accepted Risk"`.

## 6. Points de Sécurité

### 6.1 Points forts

| Point | Détail |
|---|---|
| Injection LDAP | `escape_filter_chars()` protège le filtre de recherche AD |
| Pas de mdp en base | Authentification 100% déléguée à l'AD |
| Double bind LDAP | Service account pour chercher, user pour vérifier le mdp |
| Exceptions catchées | Toute erreur AD retourne `None` sans exposer les détails |
| `is_active` check | Possibilité de désactiver un compte sans suppression |
| JWT signé | Flask-JWT-Extended gère la signature cryptographique |
| Isolation scope | `scope_filter()` garantit le cloisonnement des données |

---

### 6.2 Vulnérabilités identifiées

| Niveau | Problème |
|---|---|
| 🔴 CRITIQUE | Token stocké dans `localStorage` → vulnerable au vol par XSS |
| 🔴 CRITIQUE | Cookie JWT écrasé sans `HttpOnly` par le JS de `login.html` |
| 🟠 ÉLEVÉ | Pas de révocation JWT : `logout` ne suffit pas |
| 🟠 ÉLEVÉ | Aucun rate limiting sur `/api/auth/login` (brute force possible) |
| 🟡 MOYEN | Claims JWT non rafraîchis si rôle change en AD/DB |
| 🟡 MOYEN | Absence de protection CSRF sur les endpoints avec cookie |
| 🟡 MOYEN | `AD_USE_SSL` non forcé en configuration |

---

### 6.3 Corrections recommandées

1. **Supprimer `localStorage.setItem('voc_token')`** → reposer sur le cookie HttpOnly uniquement
2. **Supprimer le `document.cookie` JS** dans `login.html` → laisser Flask seul setter le cookie
3. **Ajouter `flask-limiter`** : 5 tentatives / minute / IP sur `/api/auth/login`
4. **Implémenter une blacklist JWT** (JTI en Redis ou DB) pour la révocation réelle
5. **Forcer `AD_USE_SSL=True`** en production avec validation du certificat
6. **Ajouter la protection CSRF** via `flask-wtf` ou le mécanisme Flask-JWT-Extended
